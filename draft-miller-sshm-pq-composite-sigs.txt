           MLDSA44-Ed25519-SHA512 Composite Signatures in SSH

Abstract

   This document specifies the integration of the MLDSA44-Ed25519-SHA512
   composite signature scheme into the Secure Shell (SSH) protocol.
   This scheme combines the post-quantum Module-Lattice-Based Digital
   Signature Algorithm (ML-DSA) with the traditional Ed25519 signature
   algorithm to provide security against both quantum and classical
   adversaries.

1.  Introduction

   The Secure Shell (SSH) protocol [RFC4251] is a protocol for secure
   remote login and other secure network services over an untrusted
   network.

   The advent of quantum computing poses a threat to traditional
   asymmetric cryptography.  Post-quantum algorithms like ML-DSA
   [FIPS204] have been developed to mitigate this threat.  However,
   during the transition period, it is desirable to use hybrid schemes
   that combine post-quantum and traditional algorithms to protect
   against both quantum attacks and potential vulnerabilities in newly
   standardized post-quantum schemes.

   This document specifies a composite signature scheme for SSH, mapping
   the COMPSIG-MLDSA44-Ed25519-SHA512 scheme defined in
   [I-D.ietf-lamps-pq-composite-sigs] to the SSH protocol [RFC4253].

3.  The ssh-mldsa44-ed25519@openssh.com Algorithm

   The algorithm name for this composite signature scheme is "ssh-
   mldsa44-ed25519@openssh.com".

3.1.  Public Key Format

   The "ssh-mldsa44-ed25519@openssh.com" public key has the following
   format:

       string    "ssh-mldsa44-ed25519@openssh.com"
       string    composite_public_key

   The "composite_public_key" string consists of the raw concatenation
   of the ML-DSA-44 public key (1312 bytes) and the Ed25519 public key
   (32 bytes), for a total of 1344 bytes.

       byte[1312] mldsa_pk
       byte[32]   ed25519_pk

   The format of the ML-DSA-44 public key is defined in [FIPS204].  The
   format of the Ed25519 public key is defined in [RFC8032].

3.2.  Signature Format

   Signatures have the following format:

       string    "ssh-mldsa44-ed25519@openssh.com"
       string    composite_signature

   The "composite_signature" string consists of the raw concatenation of
   the ML-DSA-44 signature (3309 bytes) and the Ed25519 signature (64
   bytes), for a total of 3373 bytes.

       byte[3309] mldsa_sig
       byte[64]   ed25519_sig

3.3.  Signature Generation

   To sign a message M, the signer first constructs a message
   representative M' as follows:

       M' = Prefix || Label || len(ctx) || ctx || SHA512(M)

   Where:

   *  Prefix: The ASCII string "CompositeAlgorithmSignatures2025" (43 6F
      6D 70 6F 73 69 74 65 41 6C 67 6F 72 69 74 68 6D 53 69 67 6E 61 74
      75 72 65 73 32 30 32 35).

   *  Label: The ASCII string "COMPSIG-MLDSA44-Ed25519-SHA512" (43 4F 4D
      50 53 49 47 2D 4D 4C 44 53 41 34 34 2D 45 64 32 35 35 31 39 2D 53
      48 41 35 31 32).

   *  len(ctx): A single byte representing the length of the context
      string.  For SSH, this MUST be 0.

   *  ctx: The context string.  For SSH, this MUST be the empty string.

   *  SHA512(M): The SHA-512 hash of the original message M.

   The signer then computes:

   *  mldsa_sig = ML-DSA-44.Sign(mldsa_sk, M', ctx=Label)

   *  ed25519_sig = Ed25519.Sign(ed25519_sk, M')

   The final "composite_signature" is the concatenation of mldsa_sig and
   ed25519_sig.

3.4.  Signature Verification

   To verify a signature, the verifier reconstruct M' using the same
   process as defined for generation.  The signature is valid if and
   only if both underlying signature verifications succeed:

   *  ML-DSA-44.Verify(mldsa_pk, M', mldsa_sig, ctx=Label)

   *  Ed25519.Verify(ed25519_pk, M', ed25519_sig)

4.  Private Key Format

   When adding a key to an agent [I-D.ietf-sshm-ssh-agent] using the
   SSH_AGENTC_ADD_IDENTITY or SSH_AGENTC_ADD_ID_CONSTRAINED messages,
   the key data MUST have the following format:

       string    "ssh-mldsa44-ed25519@openssh.com"
       string    composite_public_key
       string    composite_private_key
       string    comment

   Where "composite_public_key" is as defined in the Public Key Format
   section, and "composite_private_key" is the raw concatenation of the
   ML-DSA seed (32 bytes) and the Ed25519 seed (32 bytes), for a total
   of 64 bytes.

       byte[32]  mldsa_seed
       byte[32]  ed25519_seed

5.  Security Considerations

   The security of this composite scheme depends on the strength of both
   component algorithms.  An adversary must break both ML-DSA-44 and
   Ed25519 to forge a signature.

   Implementations MUST NOT reuse component key material between
   composite and non-composite keys, or between multiple composite keys.

   The use of a unique Label and the inclusion of the algorithm prefix
   in M' provide non-separability, preventing component signatures from
   being removed and used in other contexts.

6.  IANA Considerations

   IANA is requested to register the algorithm name "ssh-
   mldsa44-ed25519@openssh.com" in the "Public Key Algorithm Names"
   registry.
